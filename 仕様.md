アプリケーション開発仕様書：Thought Sublimator (思考昇華システム)
1. 背景
プロダクト開発やQA作業において、要求や指摘事項は常に明確なわけではない。「なんとなくの理想」や「言葉にできない違和感」を切り捨てず、それらが具体的な要件や忌避事項へと「昇華（進化）」していくプロセスを管理するための専用ツールが必要である。
本アプリは、4つの区分による思考整理と、項目が形を変えていく「系譜」を記録することを目的とする。

2. 思考フレームワーク（4つの区分）
全ての項目は以下のいずれかに分類される。

① 絶対にこうあるべき (Goals/Requirements): 明確なゴール、仕様、要件。

② こうだったら嬉しいなのイメージ (Nice-to-Have): ゆるやかな方針や理想。具体化により①や④へ変化する。

③ なんか違う、違和感がある (Intuitive Discomfort): 正体不明の違和感。言語化が進むと①〜④へ昇華される。

④ 絶対アカン (Critical Avoidance): 徹底して避けるべき状態。最優先で対処し、回避策を①〜③へ昇華させる。

3. 主要機能（必要な要素）
3.1 プロジェクト管理
トップページでプロジェクトを作成・管理できる。

データはブラウザの localStorage に保存し、リロードしても保持される。

3.2 メインボード (看板形式)
画面を4つのカラム（①〜④）に分割して表示。

各カラムで新規項目の追加（タイトルのみのクイック追加）が可能。

解決済み表示トグル: * ONにすると、ステータスが「解決済み」または「昇華済み」の項目も各カラムに表示される（スタイルはグレーアウト等で区別）。

3.3 項目のライフサイクル管理
各項目は以下の3つのステータスを持つ。

Active (未解決): 現在取り組むべき事項。

Resolved (解決済み): その項目が完了し、追加アクションが不要な状態。

Sublimated (昇華済み): その項目から新しい具体的タスクが生まれた状態。

3.4 昇華（Sublimation）ロジック
項目詳細画面から「昇華」アクションを実行できる。

動作:

元の項目のステータスを Sublimated に変更。

新しい項目（子）を作成し、任意の区分（①〜④）へ配置。

1対Nの対応: 1つの項目から複数の子項目へ昇華させることも可能。

履歴保持: 子項目は親項目のIDを保持し、詳細画面から親へ辿れるようにする。

3.5 UI/UX 要素
詳細モーダル: 項目名、メモ（Markdown対応推奨）、ステータス、系譜（親・子リンク）を表示。

系譜表示 (Lineage): 「この項目の由来（親）」および「この項目から発展したもの（子リスト）」をリンク形式で表示。

4. データ構造 (TypeScript)
TypeScript
type ItemStatus = 'active' | 'resolved' | 'sublimated';
type CategoryId = 1 | 2 | 3 | 4;

interface ThoughtItem {
  id: string;
  projectId: string;
  category: CategoryId;
  status: ItemStatus;
  title: string;
  memo: string;
  parentId: string | null;   // 昇華元のID
  childIds: string[];        // 昇華先のIDリスト
  createdAt: number;
  resolvedAt?: number;       // 解決または昇華した日時
}
5. 技術スタック・構成案
Frontend: React, TypeScript, Tailwind CSS

State Management: React Context API または シンプルな useState

Storage: LocalStorage (将来的な拡張性を考慮し、データ操作は抽象化しておく)

Deployment: GitHub Pages (GitHub Actions による自動デプロイ設定を含む)

6. 注意点（実装上の制約）
シンプルさの維持: 複雑なタスク管理ツール（Jira等）を目指すのではなく、直感的な思考の整理に特化すること。

データ整合性: 項目を削除する際、系譜（親・子関係）が壊れないよう配慮すること（または削除ではなくアーカイブを推奨）。

モバイル対応: スマートフォンからも「違和感」を即座に入力できるよう、レスポンシブデザインとすること。

昇華の明示: 昇華済み項目がボードを埋め尽くさないよう、トグルスイッチによる表示制御を確実に実装すること。
